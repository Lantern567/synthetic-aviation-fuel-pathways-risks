# 约束冗余分析报告

> 分析日期：2024-12-24
> 分析对象：`coal_hydrogen_optimization_model.py`
> 问题背景：模型求解过程中发生 Out of Memory 错误

---

## 1. 问题概述

### 1.1 内存溢出现象

模型在 MIP 分支定界求解阶段发生内存溢出：
- 探索了 99 个节点后 OOM
- 生成了 67,000+ 个切割平面
- 运行时间约 1.6 小时

### 1.2 模型规模

| 指标 | 数值 | 评估 |
|-----|------|-----|
| 连续变量 | 388,510 | 较大 |
| 整数变量（二进制） | 220 | 不多 |
| **约束数** | **801,643** | **过大** |
| 非零元素 | 11,426,834 | 很大 |
| 矩阵系数范围 | [8e-03, 2e+08] | 跨度过大 |

### 1.3 结论

**约束数量是内存溢出的主要原因**。经分析，约有 30-40 万约束存在冗余，可以通过优化手段消除。

---

## 2. 冗余约束详细分析

### 2.1 非负约束冗余（约 15 万个）

#### 问题描述

当前代码使用 `addConstr(var >= 0)` 添加非负约束，但 Gurobi 变量创建时可以直接设置下界 `lb=0`，无需单独约束。

#### 涉及约束

| 约束名称模式 | 估算数量 | 代码位置 |
|------------|---------|---------|
| `h2_inventory_nonnegative_{location}_{hour}` | ~7.5万 | 第 4914-4919 行 |
| `h2_inventory_nonnegative_mtj_{location}_{hour}` | ~1万 | 第 4972-4977 行 |
| `co2_inventory_nonnegative_{saf_loc}_{hour}` | ~7.5万 | 第 8319-8330 行 |

#### 当前代码示例

```python
# 氢气库存非负约束 - 冗余
for hour in range(self.total_hours + 1):
    self.model.addConstr(
        self.hydrogen_storage_vars[(location, hour)] >= 0,
        name=f"h2_inventory_nonnegative_{location}_{hour}"
    )

# CO₂库存非负约束 - 冗余
for hour in range(self.total_hours + 1):
    if (saf_loc, hour) in self.co2_inventory_vars:
        self.model.addConstr(
            self.co2_inventory_vars[(saf_loc, hour)] >= 0,
            name=f"co2_inventory_nonnegative_{saf_loc}_{hour}"
        )
```

#### 修复方案

在创建变量时直接设置 `lb=0`：

```python
# 修复前
self.hydrogen_storage_vars[(location, hour)] = self.model.addVar(
    name=f"h2_storage_{location}_{hour}"
)

# 修复后
self.hydrogen_storage_vars[(location, hour)] = self.model.addVar(
    lb=0,  # 直接设置下界
    name=f"h2_storage_{location}_{hour}"
)
```

然后删除所有 `h2_inventory_nonnegative_*` 和 `co2_inventory_nonnegative_*` 约束。

### 2.2 小时级产能约束冗余（约 10 万个）

#### 问题描述

当前代码为每个位置、每个技术、每个小时都添加一个产能约束：

```
production[loc, tech, hour] <= facility_capacity[loc, tech]
```

但 `facility_capacity` 是决策变量，在整个时间周期内值固定。每小时添加一个约束完全冗余。

#### 涉及约束

| 约束名称模式 | 估算数量 | 代码位置 |
|------------|---------|---------|
| `capacity_{location}_{tech}_{hour}` | ~10万 | 第 4443-4450 行 |

#### 当前代码

```python
def _add_production_capacity_constraints(self):
    for location in self.locations:
        for tech in self.technologies:
            for hour in range(self.total_hours):  # 遍历每个小时
                if (location, tech, hour) in self.production_vars:
                    self.model.addConstr(
                        self.production_vars[(location, tech, hour)] <=
                        self.facility_capacity_vars[(location, tech)],
                        name=f"capacity_{location}_{tech}_{hour}"
                    )
```

#### 修复方案

**方案 A：使用变量上界（推荐）**

在创建 `production_vars` 时，通过回调或后处理设置上界。但由于上界是变量，需要保留约束。

**方案 B：使用聚合约束**

将小时级约束改为时段级（如每周一个约束）：

```python
# 每周添加一个最大生产约束
for week in range(self.time_horizon_weeks):
    week_start = week * self.periods_per_week
    week_end = (week + 1) * self.periods_per_week

    max_production_in_week = gp.max_(
        self.production_vars[(location, tech, hour)]
        for hour in range(week_start, week_end)
        if (location, tech, hour) in self.production_vars
    )

    self.model.addConstr(
        max_production_in_week <= self.facility_capacity_vars[(location, tech)],
        name=f"capacity_{location}_{tech}_week{week}"
    )
```

**方案 C：保持现状但减少时间粒度**

将时间粒度从 3 小时改为 6 小时或 12 小时，约束数量减半或减少 75%。

### 2.4 管道单链路约束冗余（约 1.2 万个）

#### 问题描述

代码中存在两层约束：
1. **源地总出库约束**（第 5066-5101 行）：限制每个氢气源地的总出库量
2. **单链路约束**（第 5200-5211 行）：限制每条管道的运输量

单链路约束是冗余的，因为：
- 如果源地总出库量受限，那么单条链路的运输量自然受限
- 单链路约束增加了大量冗余计算

#### 涉及约束

| 约束名称模式 | 估算数量 | 代码位置 |
|------------|---------|---------|
| `hydrogen_source_outbound_limit_{h_loc}_weekly` | ~100 | 第 5097-5101 行 |
| `h2_pipeline_production_limit_{h2_loc}_{mtj_loc}_week` | ~1.2万 | 第 5207-5210 行 |

#### 当前代码

```python
# 约束1：源地总出库约束（保留）
self.model.addConstr(
    total_outbound_transport <= weekly_h2_production,
    name=f"hydrogen_source_outbound_limit_{h_loc}_weekly"
)

# 约束2：单链路约束（冗余）
self.model.addConstr(
    self.hydrogen_pipeline_transport_vars[(h2_loc, mtj_loc)] <= weekly_h2_production,
    name=f"h2_pipeline_production_limit_{h2_loc}_{mtj_loc}_week"
)
```

#### 修复方案

删除单链路约束 `h2_pipeline_production_limit_*`，仅保留源地总出库约束。

## 3. 优化效果预估

| 优化项 | 预计减少约束数 | 实施难度 | 优先级 |
|-------|--------------|---------|-------|
| 非负约束改用变量下界 | ~15万 | 简单 | ⭐⭐⭐⭐⭐ |
| 删除冗余管道约束 | ~1.2万 | 简单 | ⭐⭐⭐⭐ |
| 产能约束聚合/减少 | ~15万 | 中等 | ⭐⭐⭐ |

**预期效果**：约束数从 80 万降至 ~50 万，减少约 40%。

## 4. 其他优化建议

### 4.1 Gurobi 参数调整

```python
# 降低 NodefileStart，更早将节点写入磁盘
model.setParam('NodefileStart', 30)  # 从 100 降到 30 GB

# 限制切割平面数量
model.setParam('Cuts', 1)  # 或 'Cuts', 0

# 减少线程数
model.setParam('Threads', 96)  # 从 192 降到 96

# 增大 MIPGap 提前终止
model.setParam('MIPGap', 0.05)  # 从 0.01 改为 0.05
```

### 4.2 时间粒度调整

当前使用 3 小时粒度（55 期/周），考虑：
- 改为 6 小时粒度（28 期/周）：约束减少 50%
- 改为 12 小时粒度（14 期/周）：约束减少 75%

### 4.3 数值缩放

矩阵系数范围 [8e-03, 2e+08] 跨度过大，建议：
- 将单位从 kg 改为 吨 (t)
- 将成本单位从 元 改为 万元
