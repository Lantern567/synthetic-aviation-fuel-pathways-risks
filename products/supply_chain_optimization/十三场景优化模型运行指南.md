## 使用说明

### 基本格式（使用tmux）

```bash
# 创建新的tmux会话
tmux new -s <会话名称>

# 在tmux中运行命令
python <脚本路径> [参数]

# 断开会话（保持后台运行）
# 按 Ctrl+B，然后按 D

# 重新连接会话
tmux attach -t <会话名称>
```

### 为什么使用tmux而不是nohup？

| 特性 | nohup | tmux |
|------|-------|------|
| 进程隔离 | ❌ 依赖原shell | ✅ 独立服务器进程 |
| 重新连接 | ❌ 不可能 | ✅ 随时可以 |
| 查看实时输出 | ❌ 只能看日志文件 | ✅ 直接看终端 |
| 稳定性 | ❌ 可能被系统清理 | ✅ 更加稳定可靠 |

### 重要提示

1. **确保Gurobi许可证配置正确**：`/home/ljt/gurobi.lic`
2. **确保conda环境激活**：`conda activate green_methanol_for_port_transportation`
3. **会话管理**：使用 `tmux ls` 查看所有会话
4. **进程管理**：使用 `ps aux | grep python` 查看运行中的进程
5. **终止进程**：使用 `kill <进程ID>` 终止指定进程

### tmux快捷键速查

| 操作 | 快捷键 |
|------|--------|
| 断开会话 | `Ctrl+B`, `D` |
| 创建新窗口 | `Ctrl+B`, `C` |
| 切换窗口 | `Ctrl+B`, `数字` |
| 滚动查看 | `Ctrl+B`, `[` |
| 退出滚动 | `Q` |

绿氢场景使用可再生能源（风能、太阳能）通过电解水制氢。

### 1. 煤制氢+绿氢两步法 | Coal Gasification CO₂ + Green H2 Two-Step

**场景说明**：使用煤气化提供CO₂，可再生能源电解水制氢（绿氢），两步法合成SAF（甲醇路线）

**配置文件**：`products/supply_chain_optimization/coal_hydrogen_saf_optimization/data/CoalHydrogenSAFOptimizer_config.yaml`

**技术路线**：
- CO₂来源：煤气化副产CO₂（化石碳）
- 氢气来源：风能/太阳能电解水（绿氢）
- 合成工艺：CO₂ + H₂ → 甲醇 → SAF

```bash

tmux new -s coal_green_h2_two_step

# 运行优化
python /home/ljt/code_project/green_methanol_for_port_transportation-main/products/supply_chain_optimization/coal_hydrogen_saf_optimization/src/runner/unified_optimizer_runner.py \
  --process-type two_step

```

**输出目录**：`coal_hydrogen_saf_optimization/results/`

**聚类文件**：`clustering_results.json` (绿氢聚类)

**重要说明**：
- 此场景使用绿氢（可再生能源电解），CO₂利用不计入负排放（因CO₂来自化石燃料）
- 与副产氢场景使用不同的配置文件（场景8使用 `shared/data/CoalByproductHydrogenSAFOptimizer_config.yaml`）

## 通用参数说明

### 支持命令行参数的模块

以下模块使用统一的`unified_optimizer_runner.py`，支持以下参数：

- **Coal Hydrogen** (煤制氢)
- **DAC Hydrogen** (DAC制氢)
- **Green Hydrogen** (绿氢)

#### 参数详解

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `--process-type` 或 `--process` | `two_step` | 工艺类型选择 |
| `--threads` | `None` (自动检测) | CPU线程数，自动检测为cpu_count-2 |
| `--time-limit` | `10800` | 求解时间限制（秒），3小时 |
| `--mip-gap` | `0.01` | MIP间隙容差，1%表示接受1%的次优解 |
| `--log-level` | `INFO` | 日志级别 (DEBUG/INFO/WARNING/ERROR) |

**注意**：通常不需要手动指定`--threads`、`--time-limit`、`--mip-gap`，使用默认值即可。

#### 工艺类型选择值

| 模块 | 支持的 --process-type 值 |
|------|--------------------------|
| Coal Hydrogen | `two_step`, `one_step`, `byproduct_two_step` |
| DAC Hydrogen | `two_step`, `one_step`, `byproduct_two_step`, `byproduct_one_step` |
| Green Hydrogen | `two_step`, `one_step`, `byproduct_two_step`, `byproduct_one_step` |

### 不支持命令行参数的模块

以下模块使用独立的Python脚本，不支持命令行参数配置：

- **Natural Gas Two-Step**: `run_optimization.py`
- **Natural Gas One-Step**: `src/core/natural_gas_optimization_model_one_step.py`
- **Natural Gas Byproduct Two-Step**: `run_byproduct_two_step.py`

**注意**：天然气模块不包含副产氢一步法场景，因为一步法工艺不需要外部氢气输入，无法利用副产氢。

这些脚本的配置参数（如时间窗口、线程数等）需要在代码中直接修改。
默认均为4周版本，关键位置如下：
- `run_optimization.py`：`time_horizon_weeks=4`
- `src/core/natural_gas_optimization_model_one_step.py`：`time_horizon_weeks=4`
- `NaturalGasByproductHydrogenOptimizer_config.yaml`：`time_horizon_weeks=4`

## 常见问题

### Q1: 如何管理tmux会话？

```bash
# 查看所有会话
tmux ls

# 连接到指定会话
tmux attach -t <会话名称>

# 杀死指定会话
tmux kill-session -t <会话名称>

# 杀死所有会话
tmux kill-server

# 在会话内滚动查看历史输出
# 按 Ctrl+B, [ 进入滚动模式
# 使用方向键或Page Up/Down滚动
# 按 Q 退出滚动模式
```

### Q2: 如何终止正在运行的优化？

**方法1：在tmux会话内终止**
```bash
# 连接到会话
tmux attach -t <会话名称>

# 按 Ctrl+C 终止进程
```

**方法2：直接杀死进程**
```bash
# 查找进程ID
ps aux | grep unified_optimizer_runner.py

# 终止进程
kill <进程ID>

# 强制终止
kill -9 <进程ID>
```

### Q4: 内存不足错误怎么办？

1. 减少时间窗口（修改代码中的`time_horizon_weeks`参数）
2. 增加系统交换空间
3. 使用更宽松的MIP间隙（如 `--mip-gap 0.05`）
4. 减少并行线程数（如 `--threads 64`）
5. **不要同时运行多个优化进程**

### Q6: 聚类文件命名规则是什么？

**自动检测和命名机制**：

所有模块（Coal、DAC、Green H2）的氢气聚类优化器都支持**自动检测副产氢数据**并重命名输出文件：

1. **检测规则**：
   - 检查聚类成员位置名称
   - 如果包含 `steel_`（钢铁厂）、`refinery_`（炼油厂）或 `byproduct`（副产）关键词
   - 自动判定为副产氢数据

2. **命名规则**：
   - **可再生能源氢气聚类**：`clustering_results.json`
   - **副产氢聚类**：`byproduct_clustering_results.json`

3. **配置参数**：
```yaml
clustering_parameters:
  enable_clustering: true
  export_clustering_results: true
  clustering_output_path: "clustering_results.json"  # 基础文件名
  # 如果检测到副产氢，自动重命名为 byproduct_clustering_results.json
```

**示例**：
```python
# 可再生能源数据 (solar_plant_1, wind_farm_2, ...)
→ 输出: clustering_results.json

# 副产氢数据 (steel_Angang, refinery_Sinopec, ...)
→ 输出: byproduct_clustering_results.json
```

**重要提示**：
- 不需要手动修改配置文件来区分副产氢和可再生能源
- 只需要在数据源中使用正确的位置命名规范
- 优化模型会自动加载对应的聚类文件

**修复1：煤制氢模块绿氢与副产氢配置文件混淆**

**问题**：
- 运行 `--process-type two_step` 时期望使用绿氢（可再生能源），但实际使用了副产氢配置
- 配置文件映射错误，所有工艺类型都指向了 `CoalByproductHydrogenSAFOptimizer_config.yaml`
- 导致可视化结果显示负碳（来自老的配置），但新配置应该不计算负碳

**原因**：
- `unified_optimizer_runner.py` 的 `CONFIG_MAPPING` 字典错误
- 未区分绿氢场景和副产氢场景的配置文件

**解决方案**：
修改配置文件映射为：
```python
CONFIG_MAPPING = {
    'two_step': 'products/supply_chain_optimization/coal_hydrogen_saf_optimization/data/CoalHydrogenSAFOptimizer_config.yaml',  # 绿氢
    'one_step': 'products/supply_chain_optimization/coal_hydrogen_saf_optimization/data/CoalHydrogenSAFOptimizer_config.yaml',   # 绿氢
    'byproduct_two_step': 'shared/data/CoalByproductHydrogenSAFOptimizer_config.yaml',  # 副产氢
}
```

**配置文件区别**：
- `CoalHydrogenSAFOptimizer_config.yaml`（绿氢）：
  - 数据源：`solar_generation` 和 `3hourly_generation`（可再生能源）
  - 输出目录：`results/`
  - 聚类文件：`clustering_results.json`

- `CoalByproductHydrogenSAFOptimizer_config.yaml`（副产氢）：
  - 数据源：`industrial_byproduct_hydrogen` 和 `byproduct_h2_generation`
  - 输出目录：`results/byproduct_hydrogen/`
  - 聚类文件：`byproduct_clustering_results.json`

**影响场景**：
- 场景1：煤制氢+绿氢两步法 ✅ 现在使用正确的绿氢配置
- 场景8：副产氢+煤气化CO₂两步法 ✅ 继续使用副产氢配置

**修复文件**：
- `coal_hydrogen_saf_optimization/src/runner/unified_optimizer_runner.py`

**验证方法**：
运行绿氢场景后，检查：
1. 日志中显示加载的配置文件路径
2. 结果保存在 `results/` 目录（非 `results/byproduct_hydrogen/`）

**修复2：绿氢场景碳排放计算中的负碳问题**

**问题**：
- 可视化显示煤制氢场景有负碳（-46.12M kg），但配置明确禁用了负碳计算
- 实际是可视化读取了旧的结果文件（2025-11-20）

**原因**：
- 可视化脚本读取最新文件时，绿氢场景的最新结果还是11月20日的
- 那时配置可能还启用了负碳计算（或使用了错误的配置）

**解决方案**：
1. 修正配置文件映射（见修复1）
2. 重新运行绿氢场景生成新结果
3. 新结果会正确显示 `co2_utilization_credit: 0.0`

**碳核算说明**：
- 煤气化CO₂来自化石燃料（煤炭），属于化石碳
- 将化石碳循环利用制成SAF，不属于"碳移除"
- 符合国际碳核算标准（CORSIA），不应计入负排放
- 只有DAC（直接空气捕获）场景才有真正的负碳

**修复2：氢气聚类方法参数不匹配**

**问题**：调用 `cluster_hydrogen_plants()` 方法时传入 `source_type` 参数，但方法签名不接受此参数，导致报错：
